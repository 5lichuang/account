# 网页登录会话（Cookie）余额接入

## 适用范围

适用于没有公开余额 API，但登录后的网页会调用只读余额接口的上游。运营人员在自己的浏览器中完成账号、密码和验证码登录，本项目只复用已登录会话查询余额，不自动登录，也不执行充值、消费、导出或其他写操作。

需要录入四项信息：

- 上游名称：用于看板识别账号。
- 服务地址：请求 URL 的协议、域名和可选公共前缀，例如 `https://www.example.com`。
- 接口路径：以 `/` 开头的余额接口路径，例如 `/api/user/balance`。
- 登录会话 Cookie：浏览器请求头中 `Cookie:` 后面的完整值。

服务端请求形式：

```http
GET {服务地址}{接口路径}
Cookie: {登录会话 Cookie}
loginWay: 0
Accept: application/json
```

请求不自动跟随 301、302 等重定向，避免 Cookie 被转发到其他站点。修改服务地址或接口路径时，必须同时重新输入完整 Cookie；旧 Cookie 不会自动复用到新连接。

## 获取连接信息

1. 在自己的浏览器中登录目标站点，手动完成验证码。
2. 打开开发者工具的 Network，进入或刷新余额、充值或用量页面。
3. 找到返回余额 JSON 的 GET 请求，在 Request URL 中区分服务地址和接口路径。
4. 在 Request Headers 中复制 `Cookie` 后面的值，不要复制 `Cookie:` 前缀。
5. 在本项目选择“网页登录会话（Cookie）”，录入以上信息并提交。
6. 确认同步成功、余额口径正确，并观察最后成功同步时间。

Cookie 等同于登录凭证。只能粘贴到可信本机运行的项目页面，不要发送到聊天，不要写入源码、`.env`、README、日志、提交记录或截图。

## 当前响应格式

### 标准余额协议

```json
{
  "success": true,
  "data": {
    "username": "operator@example.com",
    "quota": 100000,
    "used_quota": 25000,
    "request_count": 1200,
    "currency": "CNY",
    "balance_amount": 75,
    "used_amount": 25,
    "subscriptions": []
  }
}
```

### 充值余额与馈赠金点数协议

```json
{
  "code": 200,
  "success": true,
  "data": {
    "personalRecharge": 4396013412513,
    "systemGift": 5677999022860,
    "subBalances": []
  }
}
```

换算规则：

```text
充值余额   = personalRecharge / 100,000,000
馈赠金     = systemGift / 100,000,000
可用总余额 = (personalRecharge + systemGift) / 100,000,000
```

`subBalances` 是两类余额的拆分数据，不重复累加。若某个品牌返回其他字段或不同单位，当前版本不会猜测其金额，应先取得脱敏 JSON 样例并新增明确的解析格式。

## 同步和错误处理

- 页面可见时约每 60 秒触发一次同步；这不是服务端 24×7 定时任务。
- 同步失败时保留最近一次成功余额，并更新错误状态和最后尝试时间。
- HTTP 401、HTTP 403 或业务码 `40001` 视为会话失效，需要重新登录并更新 Cookie。
- HTTP 429 表示请求过于频繁；HTTP 3xx 表示接口发生重定向；HTTP 5xx 表示上游余额服务暂时不可用。
- 上游原始错误详情和 Cookie 不返回给浏览器。

## 当前安全边界

- Cookie 和账号配置只保存在服务进程内存，服务重启或重新构建后需要重新录入。
- 真实上游必须使用 HTTPS；HTTP 只允许本机模拟测试。
- 项目不会自动登录、识别验证码或刷新登录会话。
- 当前项目没有访问控制和加密持久化，只适合可信本机环境，不应部署到公网。
- 需要 24×7 线上监控时，应先设计访问控制、Cookie 加密存储、服务端调度和会话续期。
