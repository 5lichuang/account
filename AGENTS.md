# 项目目标

构建一个可实际使用的多上游余额看板。服务端按 provider 使用 API Bearer Token、网页访问令牌、云厂商 AK/SK 签名或通用网页登录会话拉取余额，浏览器只访问本项目的脱敏接口。

# 范围与安全

- 第一版支持通用 Bearer 余额协议、网页访问令牌（Bearer＋用户标识请求头）、阿里云 BSS `QueryAccountBalance` 和可信本地环境中的通用网页登录会话（Cookie）余额查询，并支持新增、编辑、停用、删除、余额预警和同步状态。
- 上游 API Key、网页 Access Token、附加用户请求头值、AccessKey ID、AccessKey Secret 与网页登录 Cookie 只允许在服务端处理；浏览器最多接收脱敏后的凭证标识，完整凭证不得写入日志或提交到仓库。
- 网页访问令牌上游允许配置 HTTPS 服务地址、接口路径、用户标识请求头名称、点数换算除数和币种；完整 Access Token 与用户请求头值只保存在服务端。修改服务地址或接口路径时必须重新输入 Access Token，第一版不自动登录或续期。
- Cookie 会话上游允许配置 HTTPS 服务地址和接口路径，但修改任一连接信息时必须同时重新输入完整 Cookie，避免旧会话被发送到新站点；请求不自动跟随重定向，验证码与登录过程必须由运营人员手动完成。
- 不实现充值、续费、退款、外部通知等上游写操作。
- 修改持久化结构前先补充迁移方案和验证；真实密钥使用运行时 Secret，不写入 `.env` 或源码。
- 应用采用首次启动创建唯一管理员的模式，不开放普通用户注册；页面及全部管理 API 必须在服务端验证登录，`/healthz` 除外。
- 管理员密码使用 scrypt 哈希；浏览器只保存带 `HttpOnly`、`SameSite=Lax` 的会话 Cookie，服务端只持久化会话令牌摘要并设置固定过期时间。
- 第一阶段只持久化管理员与登录会话；上游配置和凭证继续保存在单进程内存，未经单独迁移方案不得写入数据库。

# 工程约定

- 使用项目初始化器生成的 TypeScript、React 与 Vite/Vinext 结构，不擅自替换框架。
- UI 和 API 字段使用英文命名，面向运营人员的界面文案使用中文。
- 优先使用现有依赖和 Web 标准，保持单页产品和最小必要后端接口。
- 修改后至少运行生产构建，并用模拟上游验证多账号、失败、空数据、无限额度，以及阿里云签名、网页访问令牌附加请求头与点数换算、Cookie 会话多格式余额换算、会话失效、连接变更凭证绑定和凭证脱敏场景。

# 腾讯云部署约定

- 当前版本只允许单机、单进程私有部署；管理员和登录会话持久化到独立 SQLite 数据卷，上游账号和凭证仍保存在进程内存，服务重启、服务器重启或重新发布后必须重新录入。
- 目标腾讯云实例使用现有 Docker，不安装全局 Node.js；生产容器必须使用 Node.js `22.13.0` 或更高版本，以单容器运行并只映射 `127.0.0.1:3210`，腾讯云安全组不得开放 3210 端口。
- 在上游凭证加密持久化和服务端调度完成前，默认通过 SSH 隧道访问。接入公网前必须具备 HTTPS，且登录保护需覆盖页面及全部管理 API。
- 生产部署只运行一个实例，不使用 PM2 cluster、多进程或多副本，避免内存账号状态分裂。
- 发布包、systemd 配置、部署文档和日志中不得包含真实 API Key、Access Token、AK/SK、Cookie、服务器密码或其他登录凭证。
- 部署验证至少覆盖 `/healthz`、容器非 root 与只读文件系统、应用仅映射 loopback、Docker 自动重启、发布失败自动回滚，以及发布包不包含 `.env`、`node_modules`、构建日志或本地凭证。
